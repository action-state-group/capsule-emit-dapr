# capsule-emit-dapr

A Go adapter that turns a **Dapr 1.18 signed workflow history** into an
**Agent Action Capsule (AAC)**, per `draft-mih-scitt-agent-action-capsule-02`.

This exists to make a mapping proposal to the Dapr/Diagrid maintainers
concrete: instead of describing the mapping in prose, this is a working
adapter that verifies a signed history's hash chain and Ed25519 signatures,
then projects it into a well-formed, verifier-passing Capsule.

```
go test ./...
go run ./cmd/capsule-emit-dapr -dir examples/happy-path
```

No Dapr cluster, no adjacent checkout, and no configuration required â€” the
fixtures and their signing keys are generated by this repository. Read the
LIMITATIONS section before drawing conclusions from that.

## LIMITATIONS â€” read this first

**This PoC has never been run against a live Dapr 1.18 cluster or a real
Dapr state store.** It is built entirely against the *public documentation*
of the workflow-history-signing feature
(`https://v1-18.docs.dapr.io/developing-applications/building-blocks/workflow/workflow-history-signing/`),
exercised only against synthetic fixtures this repository generates itself.
Nothing here should be read as "validated against Dapr" â€” it is "consistent
with what the docs say Dapr does," which is a materially weaker claim.

Specifically:

- **The protobuf shapes are modeled, not imported.** `daprhistory.HistoryEvent`,
  `HistorySignature`, and `CertEntry` are hand-written Go structs that carry
  what the documented signing algorithm and this adapter's mapping need.
  This module has **no protobuf dependency** (by design â€” see the task this
  PoC was built against). The real `HistoryEvent` is a oneof over many
  durabletask event kinds, defined in the `durabletask-go` dependency, which
  this PoC does not vendor and has not inspected.
- **Event byte-encoding is substituted.** The docs specify that events are
  "deterministically marshaled"; they don't say with what. This PoC uses
  Go's `encoding/json` (declaration-order struct fields, sorted map keys â€”
  genuinely deterministic for our purposes) as a stand-in. See "Judgment
  calls" below.
- **No cert-chain-to-root validation is wired into the CLI.** `daprhistory.VerifyOptions.TrustRoots`
  exists and `VerifyChain` will use it if supplied, but `cmd/capsule-emit-dapr`
  never populates it, so by default only the signature math is checked, not
  that certs chain to a Sentry CA bundle.
- **Batch boundaries, the definition of "the previous signature's digest,"
  and the cert-table indexing scheme are judgment calls**, not facts read
  off a spec â€” see "Judgment calls / open questions for Dapr maintainers"
  below. Several are the same open questions raised in the companion mapping
  document (`../DAPR-AAC-MAPPING.md`) at the top of this repository tree.

If you are a Dapr/Diagrid maintainer reading this: please treat every
"ASSUMPTION" and "judgment call" comment in `daprhistory/types.go` and
`daprhistory/chain.go` as a question, not a claim.

## What it does

1. **`daprhistory/`** â€” Go types mirroring the documented `history-NNNNNN` /
   `signature-NNNNNN` / `sigcert-NNNNNN` / `ext-sigcert-NNNNNN` state-store
   shapes, plus `VerifyChain()`, which:
   - recomputes `eventsDigest` = SHA-256 over each signature's batch of
     events, each event length-prefixed with a big-endian `uint64`
     (`ComputeEventsDigest`);
   - walks the hash chain, recomputing `signingInput = SHA-256(previousSignatureDigest || eventsDigest)`
     at each link (`SigningInput`);
   - verifies each link's Ed25519 signature against the certificate its
     `HistorySignature.CertIndex` names, looking the cert up **per
     signature** so certificate rotation mid-chain (a later signature
     referencing a higher index than an earlier one) verifies correctly;
   - optionally verifies each referenced cert chains to a supplied root CA
     pool (`VerifyOptions.TrustRoots`).
2. **`aacexport/`** â€” `FromSignedHistory(...)` implements the Daprâ†’AAC
   mapping (below). It calls `VerifyChain` first and refuses to export
   (returns an error, no capsule) if the chain does not verify.
3. **`anchor/`** â€” `Anchor(ctx, capsuleID, cfg)` POSTs `{"capsule_id": "..."}`
   to a SCITT-style anchor's `/v1/digest` endpoint (matching the
   `capsule-anchor` reference implementation's simple digest surface). The
   URL is configurable; a single `context.WithTimeout` bounds both the HTTP
   fetch and the subsequent (size-bounded) body read. Anchoring failure is
   always non-fatal and reported as a `Result{Anchored: false, Err: ...}` â€”
   the capsule remains a complete, valid, `self_attested` Agent Action
   Capsule whether or not anchoring succeeds.
4. **`cmd/capsule-emit-dapr/`** â€” a CLI: `-dir <fixture-dir>` reads a directory
   of state-store-shaped fixture files (one JSON file per state-store key â€”
   see "Fixture file format" below), verifies the chain, and prints the
   resulting capsule as JSON on stdout. `-anchor` optionally attempts
   anchoring (printed to stderr, non-fatal on failure).
5. **`cmd/gen-example-fixtures/`** â€” regenerates the example fixture
   directories under `examples/` (fresh keys/certs every run; for
   demonstration only).
6. **Tests** using synthetic Dapr-shaped fixtures with real Ed25519 keys and
   real self-signed X.509 certs carrying a SPIFFE URI SAN
   (`internal/testfixture`). See "Test coverage" below.

The `dapr_workflow` payload member (a top-level extension field, committed to
`capsule_id`, ignored by Class-1 verification, per Â§Extensibility â€” "all
extension points are payload-only") carries:

```json
"dapr_workflow": {
  "chain_head_digest": "<64-hex>",
  "event_count": "4",
  "signature_count": "1",
  "cert_count": "1",
  "spiffe_id": "spiffe://example.org/ns/default/order-agent",
  "trust_domain": "example.org",
  "workflow_name": "BookVenue"

```

All counts are exact decimal **strings**, never JSON numbers â€” the profile
forbids JSON floats and unsafe integers in any digest-bearing field (Â§5.1),
and `canonical.ComputeCapsuleID` errors on either.

### Why `action_type: "fyi"`, and no `disposition` block

The spec's gloss on `action_type` is *"fyi" (informational)* or *"decide"
(a disposition was required)*. A Dapr workflow's completion is not, in the
AAC sense, a disposition: there is no approver and no accept / reject /
needs_input / deferred decision anywhere in a Dapr history. A signed history
records **execution**, not a decision.

So this adapter emits `action_type: "fyi"` and no `disposition` block.
Fabricating a disposition (`approver: "policy"`, `decision: "accept"`) would
assert something the history does not contain; emitting `"decide"` *without*
one would assert that a decision was required and then decline to say how it
went. Both are overclaims.

Class 1 verification catches neither â€” `verify.go` only checks the enum, and
there is no check coupling `"decide"` to a `disposition`. **That is precisely
why the producer has to get it right.** This profile's whole thesis is the
may/did distinction and the refusal to claim more than the record supports; a
reference adapter that overclaims by default would undercut it.

Callers with out-of-band knowledge that a workflow genuinely passed a gate can
set `Config.ActionType = "decide"` and supply the matching disposition â€” but
that information does not exist in Dapr's history today. If Dapr later surfaces
human-in-the-loop approvals through Workflow, that becomes the natural
`"decide"` case, and it's the more interesting integration.

### Why `effect_attestation: "runtime_claimed"`, not `"gate_executed"`

The two seeded `effect_attestation` values are `gate_executed` ("the engine
observed the effect boundary directly") and `runtime_claimed` ("the
executing runtime asserted completion; the capsule records that claim, not
an observation"). This adapter reads an already-closed, already-signed Dapr
history **after the fact** â€” it does not sit on the Dapr execution path and
does not itself observe the effect boundary the way a policy gate would.
Grading the claim `gate_executed` would overclaim what this PoC actually
witnessed: a cryptographically verified *record* of completion, not a live
observation of it. `runtime_claimed` is the honest, conservative choice.

### Time binding

The Dapr docs state time binding is "certificate validity checked against
the event timestamp" â€” there is no external timestamping authority in the
signing scheme itself. This adapter's `timestamp` field is therefore only as
trustworthy as the signing sidecar's local clock and the cert's validity
window; it is not independently corroborated until (and unless) the capsule
is anchored to a transparency log, which is why `assurance.attestation_mode`
starts at `self_attested` and only an anchor Receipt can justify upgrading
it â€” a step this PoC's `anchor` package supports registering *toward*, but
does not itself perform the upgrade for (see `anchor` section above; the CLI
prints the capsule as `self_attested` even after a successful `-anchor`
call, on purpose â€” re-wrapping the payload with a verified Receipt is a
substrate/COSE-envelope concern, out of scope here).

## Judgment calls / open questions for Dapr maintainers

The public docs describe the signing *scheme* precisely but leave several
implementation details unspecified. Each is called out with an
`ASSUMPTION`/judgment-call comment at its point of use in the code; this is
the consolidated list. (Items 1â€“4 mirror the open questions already raised
in `../DAPR-AAC-MAPPING.md`; the rest are new, specific to building a working
verifier rather than describing the scheme in prose.)

1. **Batch boundaries.** What determines where one `HistorySignature`'s
   batch ends and the next begins â€” time, event count, or a workflow
   milestone? This PoC's `HistorySignature` carries explicit
   `from_event_index`/`to_event_index` fields and *requires* the union of
   all signatures' ranges to exactly, contiguously partition the full event
   list (no gaps, no overlaps) â€” which may be stricter than what Dapr
   actually guarantees.
2. **Chain head at workflow completion.** Is there a defined "final"
   signature for a completed workflow, or is the head simply the
   last-written index? This PoC treats "last signature by index" as
   authoritative for `effect.response_digest` / `dapr_workflow.chain_head_digest`.
3. **What does "the previous signature's digest" digest?** This PoC assumes
   `HistorySignature.PrevSignatureDigest = SHA-256(previous signature's raw
   signature bytes)`, empty for the chain's first link
   (`daprhistory.SignatureDigest`). An equally plausible reading is a digest
   of some other chain-state value (e.g. the previous signing input). This
   is the single highest-impact ambiguity in the whole PoC: get it wrong and
   every downstream signature fails to verify against a real Dapr history.
4. **`ext-sigcert-NNNNNN` / cert-table indexing.** This PoC assumes
   `sigcert-N` and `ext-sigcert-N` share one flat certificate-index space
   (a `HistorySignature.CertIndex` of `N` means "whichever of `sigcert-N` /
   `ext-sigcert-N` exists"). If Dapr in fact keeps two independently-indexed
   tables, `HistorySignature` needs a second discriminator field.
5. **Event canonicalization.** "Deterministically marshaled" â€” deterministic
   binary protobuf, `protojson` with sorted fields, or something bespoke?
   This PoC substitutes `encoding/json` over a hand-modeled struct; the real
   answer matters for anyone trying to reproduce Dapr's actual
   `eventsDigest` values byte-for-byte (this PoC's synthetic fixtures are
   self-consistent but not claimed to match real Dapr's digests).
6. **Does Dapr have, or want, a disposition to record?** This PoC emits
   `action_type: "fyi"` because a signed history contains no approver and no
   decision (see above). That's the honest reading, but it's worth confirming:
   is there anywhere in Dapr â€” Workflow, Conversation, an agent framework
   layer â€” where a human or a policy actually *disposes* of a request? That
   is the `"decide"` case, and it's the more valuable one to capture.
7. **Where does the workflow instance ID actually live?** Real Dapr keys
   the whole history under the workflow's actor/instance ID; it is not
   naturally a per-`HistoryEvent` field. This PoC models it as a per-event
   field for fixture convenience and requires it to be consistent across
   every event â€” a real integration would more likely read it from the
   state-store partition key rather than the event body.
8. **Root-of-trust validation is scoped out of the CLI.** `VerifyChain`
   supports an optional root CA pool, but nothing in this PoC fetches or
   pins the Sentry CA trust bundle; a real integration needs that wired in
   before "the signature verifies" can mean "and I trust the identity that
   made it."

## Fixture file format

This PoC's own convention, **not** part of Dapr's documented format: a
"state-store-shaped fixture directory" is a plain directory of JSON files,
one per state-store key, named by prefix:

```
history-000000.json      -- daprhistory.HistoryEvent
history-000001.json
...
signature-000000.json    -- daprhistory.HistorySignature
...
sigcert-000000.json      -- daprhistory.CertEntry (Foreign: false)
ext-sigcert-000000.json  -- daprhistory.CertEntry (Foreign: true)
```

`[]byte` fields (certificate DER, signature bytes) serialize as base64
JSON strings via Go's standard `encoding/json`.

## How to run

```sh
# From this directory:
export GOFLAGS=-mod=mod GOPROXY=off   # if module downloads are blocked; the
                                       # replace directive + stdlib is all
                                       # this module needs
go build ./...
go vet ./...
go test ./...

# Regenerate the example fixtures (fresh keys/certs each run):
go run ./cmd/gen-example-fixtures

# Export a capsule from a fixture directory:
go run ./cmd/capsule-emit-dapr -dir examples/happy-path

# ...with an operator override and an anchor attempt:
go run ./cmd/capsule-emit-dapr -dir examples/happy-path -operator my-tenant -anchor
```

`examples/happy-path/` is a single-signer, single-batch history ending in
`ExecutionCompleted`. `examples/cert-rotation/` is the same workflow signed
by two different certificates across two batches, exercising the
cert-rotation path.

## Test coverage

- `daprhistory`: happy-path chain verification; certificate rotation
  mid-chain; a tampered event breaks the chain; a tampered signature fails
  verification; a gap in batch coverage is refused; an unsupported signature
  algorithm (e.g. RSA) is rejected outright (this PoC verifies Ed25519 only).
- `aacexport`: happy path (`FromSignedHistory` â†’ stable `capsule_id` across
  repeated calls â†’ `verify.Verify(capsule, nil, regs).OK == true` with the
  real Class-1 verifier and the real `spec/REGISTRY.md`); export is refused
  on a broken/tampered chain; an in-progress (non-terminal) workflow maps to
  `effect.status: "dispatched"` with no `response_digest`, and still passes
  verification; an `operator` override is honored; the confirmed-effect
  binding is unit-tested directly (`buildEffect("confirmed", "", ...)` is
  rejected with `*ConfirmedWithoutDigestError`).
- `anchor`: a successful anchor round-trip against an `httptest` server; a
  non-200 response is reported as non-fatal; a slow-body server is cut off
  by the configured timeout (proving the timeout spans the body read, not
  just the initial fetch); a connection failure (closed port) is reported as
  non-fatal.

Run `go test ./... -v` to see every case by name.
